# Capstone Checklist

Use this checklist before calling your capstone project complete.

## Core structure

- [ ] project has clear package structure
- [ ] controller layer exists
- [ ] service layer exists
- [ ] repository layer exists
- [ ] model or entity layer exists

## Data and CRUD

- [ ] at least one entity exists
- [ ] create endpoint works
- [ ] read all endpoint works
- [ ] read by id endpoint works
- [ ] update endpoint works
- [ ] delete endpoint works

## Input quality

- [ ] validation is added
- [ ] bad input is handled clearly
- [ ] missing records are handled clearly

## Clean API design

- [ ] route names are sensible
- [ ] status codes make sense
- [ ] response format is understandable
- [ ] DTOs are used where helpful

## Project quality

- [ ] `application.properties` is present
- [ ] README exists
- [ ] run steps are written clearly
- [ ] sample request JSON is included

## Extra quality items

- [ ] logs are used sensibly
- [ ] Swagger or docs support is added
- [ ] profiles are considered
- [ ] Dockerfile added if comfortable
- [ ] basic security added if comfortable

## Presentation readiness

- [ ] you can explain each package
- [ ] you can explain the request flow
- [ ] you can explain why you used services
- [ ] you can explain why validation matters
- [ ] you can explain one improvement you would add later

## Final rule

If the project works and you can explain it clearly, you are in a strong position.
